(ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f62d5614a001Fkl3QdxzhAnTkz)

(ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f62d5614a001Fkl3QdxzhAnTkz)

## Learnings

- [2026-07-14] Executor does not expose every upstream tool as a separate MCP tool; it exposes only `execute`, `skills`, and conditional `resume`, with the agent searching/calling the catalog from sandboxed TypeScript inside `execute` — this reduces tool-schema/context load but changes agent behavior and requires quality instruction/approval handling. (ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f62d2ad05001DF38OO4PX8Sd1L)
- [2026-07-14] Compatibility matrix across the 11 servers: strong fit for `context7`, `deepwiki`, `openai-docs`, `aws-knowledge`, `x-docs`, `exa`; good for `exa-research` (keep separate if catalog filtering matters), `pinescript-docs` (local stdio only), `qmd` (probably good, cwd dependency to validate); conditional for `homeassistant` (credential in URL path); poor as a singleton for `fff` (its search root follows project/cwd, but a global service has a fixed cwd). (ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f62d5614a001Fkl3QdxzhAnTkz)
- [2026-07-14] Wrapping integrations through Executor shifts the authorization boundary from the harness to Executor: the harness only sees the `execute` call, so Executor's per-tool policies become the real safety boundary; the CLI defaults to `--elicitation-mode model` which is unsafe for write operations, so destructive integrations need `--elicitation-mode browser`. (ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f62d5614a001Fkl3QdxzhAnTkz)
- [2026-07-14] A single default Executor endpoint exposes the same catalog to all harnesses, conflicting with Mindframe-Z's current per-harness segmentation; Executor toolkits could in principle preserve this via `/mcp/toolkits/<name>`, but the Mindframe-Z schema cannot render a different URL per harness from one server declaration. (ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f62d5614a001Fkl3QdxzhAnTkz)
- [2026-07-15] Observed result: end-to-end disposable-prototype run passed — Executor exposed exactly `execute`, `skills`, and browser-gated `resume`; the `execute` sandbox searched the DeepWiki catalog and returned a live upstream answer. (ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f63ee7a530015bB570mlBzY4FC)
- [2026-07-15] Observed result: a schema-valid OpenCode local MCP entry (`executor mcp --scope <scope> --elicitation-mode browser`, 60s timeout, OpenCode v1.18.1) worked; piped `opencode run` children were found to wait for stdin EOF before executing, causing indefinite stalls, fixed by closing `child.stdin` immediately in the wrapper. (ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f6420ad17001AR2wAklzPiGNFm)
- [2026-07-15] Observed result: an isolated OpenCode agent completed the full flow — called `executor_skills`, made multiple `executor_execute` calls, self-corrected from an incorrect dotted dynamic path to bracket-path invocation (`tools[path](...)`), and returned the correct DeepWiki-backed answer. (ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f64234be7001ZlnWShOENz7j3K)
- [2026-07-15] Observed result, prototype-scoped model comparison at medium reasoning effort (clean-room isolated sessions, two tasks each): Luna passed both (19.4s, 41.2s) and Terra passed both (14.1s, 36.7s), both calling `executor_skills` then `executor_execute` without attempting `executor_resume`; Sol passed the two-execution research task (38.0s) but failed the simple discovery task (23.0s) with a correct answer but no successful upstream call, likely a model-knowledge fallback rather than genuine tool use. (ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f66a010b4001CzvZg3aDHWlOXl; ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f66b5e157001XjwBsQXFnMUtN5)

(ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f675d4db4001Qdl13HgKdFVV69)
- [2026-07-15] Observed result: after the slug fix, all five anonymous remote servers (`aws-knowledge`, `context7`, `deepwiki`, `openai-docs`, `x-docs`) registered and connected successfully; `exa`/`exa-research` (credentialed remote) and `homeassistant` (credentialed LAN) were deferred since credentials weren't read in this pass, and `fff`/`qmd`/`pinescript-docs` (local stdio) were deferred pending stdio policy. (ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f67478119001h5YJL3AF535ODK)
- [2026-07-15] Observed result, limited to tool-inventory visibility (no actual tool invocation tested): registering Pine Script's `npx -y pinescript-mcp-server` in a disposable Executor scope caused Executor to launch the local server, create `tools.pinescript-docs.org.default`, and a fresh Executor MCP host listed `pinescript-docs` in its execution guidance — but a stale/cached-guidance issue means a fresh MCP host is needed after adding a local integration, and the unpinned npx command is not recommended for broad deployment. ## Issues

(ses_09d2e782fffedYt0lhjlKyy3w3 · turn msg_f62d5614a001Fkl3QdxzhAnTkz)
