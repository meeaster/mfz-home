## Learnings

- [2026-07-14 22:48] Executor is not a transparent MCP multiplexer: it exposes only `execute`, `skills`, and conditional `resume`, forcing agents to write sandboxed TypeScript that searches Executor's catalog and invokes tools via `execute`, adding discovery hops versus direct MCP (source: `packages/hosts/mcp/src/tool-server.ts:931`, `packages/core/execution/src/skills.ts:28`). (ses_09d2e782fffedYt0lhjlKyy3w3 · msg_f62d5614a001Fkl3QdxzhAnTkz)
- [2026-07-14 22:48] Routing through Executor shifts the authoritative safety boundary: harnesses today reason about individual MCP tools, but through Executor they see one `execute` call that may invoke several inner tools, so Executor's own policies become the real safety boundary, not harness-level tool filtering. (ses_09d2e782fffedYt0lhjlKyy3w3 · msg_f62d5614a001Fkl3QdxzhAnTkz)
- [2026-07-15 09:52] Observed result: detached `daemon run` does not preserve workspace scope in Executor's CLI source, so setup calls made after starting a temp-scoped daemon could attach to a different, contaminating daemon; fixed by owning a foreground daemon child and terminating that exact process at cleanup. (ses_09d2e782fffedYt0lhjlKyy3w3 · msg_f63f02f0b0016rN1EZ7AKBbqSR)
- [2026-07-15 09:52] Observed result: Executor's daemon readiness output reports a canonicalized hostname rather than the literal `127.0.0.1`, so a launcher waiting on that literal string never sees it; fixed by querying the temp daemon endpoint directly, making readiness port-independent of the displayed hostname. (ses_09d2e782fffedYt0lhjlKyy3w3 · msg_f63f2783e001rAsTuE6ovHBgUX)
- [2026-07-15 09:52] Observed result: the prototype confirmed Executor's MCP surface is exactly `execute`, `skills`, and browser-gated `resume`, and `execute` successfully searched the DeepWiki catalog and returned a live answer with no unexpected tool exposure or approval bypass.
