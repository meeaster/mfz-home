## Decisions

- [2026-06-26 22:28] **Evolve thread-log from a skill into a CLI-driven orchestration layer**, keeping the existing skill working in parallel. Fork posed: (A) keep everything in a skill vs. (B) extract a CLI as the primary execution layer. Chose B. Why B won: a skill is locked to Claude Code, whereas a CLI gives "programmatic control over the process... more structured more deterministic" and can dispatch to Claude Code, OpenCode, or future ChatGPT; it also opens automations, webhooks, and a UI later. "this is all being built into a skill, but I'm wondering if we should kind of maybe extract this out and maybe make this more of a CLI... and then the CLI can be responsible for dispatching off the agent." (0af0f6b3 · turn 7)
- [2026-06-26 23:09] **The CLI is an orchestration layer, not a determinism engine** — an explicit correction of the mental model. "I don't think necessarily CLI is the way to deterministically do stuff for the agents. I think the CLI is... a way to orchestrate agents or to orchestrate a workflow with agents." Judgment steps stay agent dispatches; deterministic work (validation, file building, candidate-gathering) wraps around them, and the harness is choosable per step. (0af0f6b3 · turn 68)
- [2026-06-26 23:24] **Build a custom lightweight orchestrator rather than adopt an off-the-shelf workflow engine.** Fork: (A) custom in-house orchestrator vs. (B) Vercel AI SDK / LangGraph / Mastra / Eve / OpenAI Codex. Chose A after research. Why: every framework needs an API key for the *orchestrator's own reasoning* — they can shell out to a CLI as a tool, but can't drive subscription-harness agents as the reasoner. "ideally we're not gonna create a workflow engine for this... it would be nice if I can use these, but I don't necessarily know if they support my cases." (0af0f6b3 · turn 73)
- [2026-06-26 23:19] **Run dispatched research/work agents on Haiku 4.5, not Opus**, for cost control. "Could you maybe run these agents with the haiku model haiku 4.5... since I'm using Opus right now I don't want this to consume a lot of tokens." (0af0f6b3 · turn 86)
- [2026-06-26 22:47] **Accept ~$1–2 per session as the baseline ingestion cost.** ~$1.50–$2 on Opus, "probably get down to a dollar" on Sonnet; the two-stage Haiku-gather → Opus-synthesize split is the cost-optimized path and further optimization is likely diminishing. "maybe that's just the cost of using AI... the value that we get out of every session is just like well, yeah, it costs a dollar, but the fact that you have it integrated." (0af0f6b3 · turn 55)
- [2026-06-26 23:28] **Defer evals out of v1; treat them as a dev-time regression suite, not a per-run cost.** Fork: (A) build an eval framework into v1 vs. (B) defer. Chose B. Why: evals answer "did my change break synthesis?" on a separate budget and cadence — "this isn't necessarily a cost thing associated with the cost of the ingestion... this is just the cost to just validate our stuff is all working." When built, wrap the CLI with Inspect or DeepEval. (0af0f6b3 · turn 105)
- [2026-06-26 23:42] **Observability is a v1 cross-cutting requirement, not a feature.** Must thread through everything: a run registry (DB or JSON, a run keyed by PID), per-run logs, cost surfaced at every step, and a **history of which model did which action**. "we need to be able to investigate when things are going good, when things are going bad... surface costs... keep the history of whatever." (0af0f6b3 · turn 110)
- [2026-06-26 23:56] **Separate thread storage from config storage; config is public, threads are sensitive.** mindframe-z is (and should stay) public for sharing config/skills/MCPs, but thread data is sensitive (work vs. personal) and spans machines. Solution: mindframe-z holds **multiple configured backup destinations** and each thread is assigned one — work threads to a private work repo, personal threads to a personal/shared repo. No central mega-repo. "I think the thread stuff needed to be separate from the config." (0af0f6b3 · turn 115)
- [2026-06-27 00:15] **One-session-to-one-thread is a soft default, not a hard rule.** Fork: (A) hard rule — a session belongs to exactly one thread vs. (B) soft default — usually one, occasionally two. Chose B. Why: cost favors one extraction, but a session can legitimately span two threads' topics, in which case it earns **two different extractions** (different charter → different extraction lens), which raises a deferred storage question (sessions inside a thread vs. a shared `sessions/` pool). The user reframed all his stated "requirements" as non-binding: "I don't say those are hard requirements, I'm just thinking out loud." (0af0f6b3 · turn 129)
- [2026-06-27 00:23] **The existing skill stays the working path; the new core is built separately alongside it.** Zero risk to the thing that already works; observability and the new orchestration live in the new core, not retrofitted into the skill. "the system I have is kind of working already... we can continue using a skill as we develop, but then we can be building this other stuff on the side without touching the existing skill... that is our core." (0af0f6b3 · turn 134)
- [2026-06-27 00:32] **Design for an MCP server interface, but do not build it in v1.** Fork: (A) CLI-only vs. (B) also ship an MCP server. Chose design-for-defer. "the CLI is just the API essentially to interacting with this. I might end up just making an MCP server to interface with all this stuff... Let's design it with that in mind, but then let's not do it for V1." (0af0f6b3 · turn 134, 138)
- [2026-06-27 00:32] **Design for ChatGPT ingestion via a source-adapter pattern, build it in v2+.** Fork: (A) build ChatGPT ingestion in v1 vs. (B) design-for, defer. Chose B. Why: no robust ChatGPT API — would need the existing mindframe-z browser-engine project (out of scope for this thread). A ChatGPT chat "is no different than an open code session or a cloud code session" conceptually, so it fits the same adapter port; v1 scope is Claude Code + OpenCode. (0af0f6b3 · turn 48, 138)
- [2026-06-27 00:38] **The session-reader skills (`claude-code-sessions`, `opencode-sessions`) stay pure, portable readers.** Fork: (A) skills also own archival/restore/file-management vs. (B) skills are purely "how to read a session"; all archival/storage logic lives in the CLI/core. Chose B. Why: workflow-specific logic destroys portability — "they're supposed to be portable and not specific to a specific workflow." Skills may expose an *alternative source path* (read from an archive repo) but must never own the *decision* of where to archive. (0af0f6b3 · turn 147)
- [2026-06-26 22:28] **Build the system into mindframe-z rather than a separate slim repo.** Why: mindframe-z is becoming the "all-in-one" — config management, skill management, MCP management, reference management, sandboxing — and the thread system is the next module; consolidating makes the rest shareable too, even though the user briefly considered a standalone repo "so someone can take this." (0af0f6b3 · turn 7)
- [2026-06-27 00:40] **Capture this session into the new `thread-log-system` thread.** A high-rationale design session — exactly the case for max-effort synthesis. "So let's capture this session into a new thread." (0af0f6b3 · turn 152)

## Learnings

- [2026-06-26 23:24] **No off-the-shelf workflow engine can orchestrate subscription-CLI agents without an API key for its own reasoning.** Haiku research agent surveyed Vercel AI SDK, LangGraph, Mastra, Eve, and OpenAI Codex: all require an orchestrator API key; they can shell out to a CLI as a *tool node*, but the reasoning that drives the workflow still bills an API key. Claude Code's own subagent system is already a working orchestrator for the current skill. This is the evidence behind the build-custom decision. (0af0f6b3 · turn 73)
- [2026-06-26 23:23] **Eval frameworks exist, are CLI-wrappable, and don't require API keys if you bring your own model.** Haiku research agent surveyed OpenAI Evals, Braintrust, Inspect, LangSmith, promptfoo, DeepEval, and Ragas; **Inspect and DeepEval** are the strongest (custom model functions, CLI-callable via a wrapper), promptfoo the most mature/accessible. Confirms evals can be added later without architectural lock-in — supports deferring them. (0af0f6b3 · turn 73)
- [2026-06-26 22:47] **Existing economics confirmed from the prior threads:** reading the `thread-log-build` and `thread-log-usage` digests reconfirmed ~$1/session at the two-stage Haiku→Opus split, and the established single-source-of-truth + cost-consciousness principles — the foundation this CLI work builds on. (0af0f6b3 · turn 55)
- [2026-06-26 23:55] **mindframe-z is public.** Confirmed live in-session ("MindFrame Z is public. It is public.") — the fact that forces thread data into separate, private destinations rather than the main repo. (0af0f6b3 · turn 115)

## Intent & Vision

- [2026-06-26 22:28] The originating problem, verbatim: "I start up a new session and I get all the way throughout the session... doing a bunch of designing, and then I'm getting really close to my context limit, and I'm like, well, I don't have enough context left to create a summary document or design document, but if I compact I'm going to lose a bunch of information. But then I'm just like, wait a second, well, the session details are on the disk — well, why don't we just read the session details... and that way this whole thread management becomes another process thing on top of, separate from your actual models." (0af0f6b3 · turn 7)
- [2026-06-26 22:28] The memory-system stance: this is "almost like a memory system," but unlike auto-memory (where the AI pushes to memory and edits CLAUDE.md files), the design is inverted — "rather than having AI go and push stuff to the system, we will just have an AI that kind of sits on top of the system and watches what you do." The session files are the immutable source: "what's already written stays written but things can be added to it, so the append immutable source we use that, and then we go from there." (0af0f6b3 · turn 7)
- [2026-06-26 22:28] The deeper why — an evolving knowledge layer: "these threads evolve over time, that's the natural cycle of how we solve problems... and that's the goal we're trying to solve here: how do we evolve our knowledge layer, our context layer, to evolve over time with the work that we are doing." (0af0f6b3 · turn 7)
- [2026-06-26 22:28] Shared origin / motivation: the user's manager hits the same problem juggling ChatGPT, Claude Code, and OpenCode contexts — "everyone thinks they're building the right memory system for their AI agents." This is what pushes toward multi-harness, multi-source support. (0af0f6b3 · turn 7)
- [2026-06-26 22:28] The cost-lever philosophy: "I think eventually we're gonna have different levers we can pull... like, yeah, let's get everything and then have the best model synthesize it all and produce output — but then it's like, well, can we get away with maybe only returning back a subset of the information? Can we maybe use a cheaper model for synthesization?" (0af0f6b3 · turn 7)
- [2026-06-26 22:28] A context-management caution that constrains digest size: "the idea is when we go into this feature... you have a context window of 200,000 and you just loaded a hundred thousand of it in from memory — that's not really valuable. But we still want to surface up information that is valuable to what we're doing." (0af0f6b3 · turn 7)
- [2026-06-26 22:28] Isolation/sandboxing aspiration for the synthesis step: "when I go and do the synthesis, to be honest, I don't really want you to have access to anything... I want to run you in a container." mindframe-z already has sandbox work to build on, and Agent Vault is floated for proxying credentials. (0af0f6b3 · turn 7)
- [2026-06-26 22:28] Where it's heading, long term: "we'll start with a CLI for how we interact with everything, then maybe we can even move to an actual UI like a web UI or a desktop application where these threads can be managed... you can kick off synthesis." Framed as a long-term goal — "right now we need to start with baby steps." (0af0f6b3 · turn 7)
- [2026-06-26 22:28] And the automation aspiration: "in the future we could probably do automations where it's like, oh hey, you just— a new session was created — okay, we just noticed this is part of your work, do we want to integrate it into this thread? I think this would be really cool to build if we can pull this off." (0af0f6b3 · turn 7)
- [2026-06-26 22:28] On where to build it: mindframe-z as the "all-one" — "it's gonna include config management, skill management, MCP management, reference management, sandboxing... and then the next one would be this — I don't want to call it a memory system — this thread system that allows you to manage your threads." The tension: wanting to share config easily without forcing others onto his exact setup. (0af0f6b3 · turn 7)
- [2026-06-26 22:40] On scope of the change: "I'm not saying this is like a totally different system, but it's like an extension where the same process, the same workflows that happen currently we'll trigger from a CLI — most likely will be triggered from Cloud Code." (0af0f6b3 · turn 48)
- [2026-06-27 00:14] On rigor vs. exploration: "I don't say those are hard requirements, I'm just thinking out loud what I'm thinking" — the whole session is exploratory brain-dump, and the user wants brainstorming and options, not a locked spec. (0af0f6b3 · turn 129)
